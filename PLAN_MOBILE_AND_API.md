# План: мобильный клиент, API и домен

Документ-напоминание для решения позже: что уже есть, чем Flutter/React Native отличается от «сайта в телефоне», где жить коду и как это стыковать с сервером.

## Что уже сделано (эксперимент)

- В репозитории есть опциональный **`api_fastapi/`** — отдельный процесс на FastAPI (например порт **8001**).
- Страница **`/docs` (Swagger)** — **не пользовательский интерфейс**, а техническая документация и песочница для разработчика. На неё не стоит ориентироваться как на «мобильную версию».

## Два разных понятия «мобилка»

### 1) Нативное приложение (React Native / Flutter и т.п.)

- Отдельный **продукт**: свой репозиторий *или* папка в том же монорепо (`mobile/`, `apps/client/` и т.д.).
- Пользователь **ставит приложение** из магазина или APK/IPA — на сервере оно «не открывается как сайт», приложение **установлено на устройство**.
- Клиент **ходит по HTTPS** на ваш backend и забирает **JSON** (в т.ч. с FastAPI-слоя или с другого API, если решите иначе).
- **Не путать** с браузером: иконка на рабочем столе телефона — это не автоматический редирект с `https://ваш-сайт/`.

### 2) Сайт в браузере на телефоне

- Открывается **тот же Django-сайт** (те же шаблоны), если не делать специальных доработок.
- Чтобы было «удобно с телефона», нужна отдельная работа по **адаптивной вёрстке** (CSS, при необходимости разные шаблоны) или **PWA** / отдельная мобильная веб-ветка — это **не** то же самое, что нативное приложение.

**Итог:** нативное приложение ≠ «просто зашёл на сайт с телефона». Это разные дорожные карты (можно делать только одну из них или обе).

## Где хранить код мобильного приложения

| Вариант | Смысл |
|--------|--------|
| Отдельный Git-репозиторий | Проще права доступа и релизы только мобилки. |
| Тот же репозиторий (monorepo) | Одна история задач; рядом `api_fastapi/`, Django, папка `mobile/`. Артефакты всё равно разные (сборка приложения ≠ deploy сайта). |

Класть React Native / Flutter прямо «внутрь» пакета Django не обязательно и часто неудобно — достаточно **рядом** в том же mono-репо или в другом repo.

## Сервер и домен

Типичная схема в проде:

- **Сайт (Django):** nginx → Gunicorn, например `https://example.com/`.
- **API для приложения:** тот же сервер за **тем же доменом** через префикс, например `https://example.com/api/` → прокси на процесс FastAPI (или поддомен `https://api.example.com/` — тоже норма).
- Приложение из магазина **не лежит на сервере как HTML**; на сервере — API, медиа при необходимости, та же БД/интеграции, что и у сайта.

FastAPI-слой можно дорастить позже (**авторизация**, контракты эндпоинтов под экраны) или вообще не использовать, если решите брать данные только с Django через другой канал — тогда экспериментальную папку можно удалить.

## Когда будете готовы делать мобильный клиент — о чём решить заранее

1. **Чтение или запись** с телефона (только отчётность или склад/правки)?
2. **Авторизация** API (ключ, JWT, привязка к `.biota_users.json` или отдельная модель пользователей).
3. Какие **экраны** в первой итерации (список сотрудников, смены, СКУД, склад — не всё сразу).
4. Нужен ли **вообще** отдельный FastAPI или достаточно позже **DRF/endpoint’ы в Django** (тогда часть дублирования SQL может не понадобиться).
